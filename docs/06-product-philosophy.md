# 06 — 产品哲学

> 这是 v3 → v10 多次迭代后凝结下来的核心判断。
> **每次做新功能/写新代码前，先回顾这里**。它决定了"做什么"和更重要的"不做什么"。
>
> 本文档是这个产品**最稀缺的资产**——「反主流」的产品品味需要明示成文档，
> 否则任何新加入的人都会本能地往"加更多功能"的方向走。

---

## 一句话总结

**平台 = 撮合 + 关键节点 + 激励发放。不是留痕系统，不是流程管控者，更不是注意力收割机。**

用户接上头之后想怎么聊、约在哪、用什么沟通——都是 ta 的自由。
我们提供工具，**不挡用户的路 · 不强加焦虑 · 不抢夺注意力 · 不假装繁荣**。

---

## 八条核心哲学

### 1. 不挡用户的路

用户从平台拿到的价值是**找到宠物 / 归还宠物 / 善行值徽章**，不是"在平台聊天的体验"。

具体含义：

- **不介意用户私加微信**。我们不主动引导（微信审核合规要求），但也不阻止
- **沟通话术 / 安全建议**这些"教程"不强制走完。它们以"按需 modal"形式存在，主动点才看到
- **归还流程极简到一个按钮**——线下完成归还后，回来打个卡就完事
- **自检清单是建议，不是验证**。即使一项都不勾，也能确认归还。平台信任用户的判断

### 2. 反冒领的真实边界

**v10 已彻底砍掉「私有特征」字段**——它防不住代答/转述/熟人冒领，反而**误伤真实好心人**（好心人不会刻意记宠物的"独家特征"，被失主考特征时答不上来，可能误判为骗子）。

真正的防御靠：

- **宠物对失主的反应** — 真正的失主，宠物通常立刻识别（这是最可靠的）
- 疫苗本 / 旧合影 / 家里其他照片
- 公共安全地点交接（派出所、宠物医院、咖啡馆）
- **24h 归还冷却期**（详见 `02-state-machines.md`）— **这是平台层面的最后一道防线**

`contact-guide` 屏（联系建议）的提示已统一为：

> 💡 **不需要让对方先描述细节。最可靠的核对发生在见面时**——宠物对失主的反应、疫苗本、家里旧照片，比"考特征"准得多。

**演进轨迹**：v3 → v4 → v6 我们逐步把"私有特征"从"前置告知发现者"降级为"失主心里有数 + 后置仲裁依据"；v10 直接砍 —— 失主获得的"我有防御工具"安全感由「核对指南」承载，而不是一个误伤好心人的考核字段。

### 3. 关键节点必须在平台内

什么是"关键节点"？善行值发放、争议仲裁需要平台介入的所有动作。具体：

- 发布丢失/招领（数据入库）
- 提供线索（线索表入库）
- 失主标记线索"有用 / 关键"（触发善行值预结算）
- 确认归还（触发 24h 冷却 → 善行值结算）
- 写感谢信（触发被点名者善行值）
- 争议申诉（进入仲裁队列）

**这些必须在小程序内完成**，因为是平台"看到归还发生 + 发放激励 + 受理纠纷"的唯一依据。

非关键节点（聊天约见面/线下沟通/确认特征）可以**完全在平台外**，平台只需要提供入口和默认渠道（平台留言系统），不强制留人。

### 4. WeChat 合规底线（不可破）

微信小程序对"诱导加微信 / 拉新返利 / 未审核交易"盯得很紧。三条硬红线：

| 禁止 | 我们的应对 |
|------|-----------|
| UI 出现"加微信 / 留微信号 / 加好友"字眼 | 所有文案严格审查；话术模板中只说"约个地点见面"不说"加我微信" |
| 拉新奖励虚拟积分（疑似传销/返利） | M0 阶段**完全不做邀请奖励善行值**。等用户量起来再评估 |
| 平台撮合的金钱交易（悬赏/感谢金担保） | 平台**不介入任何金钱交易**。文案中说"如有感谢金请线下沟通"，不提供任何代收/担保功能 |

类目选**公益 / 动物救助**（不选"生活服务-社区"），审核标准更宽松，且和"善意网络"定位一致。

### 5. 大理冷启动的特殊性

**为什么大理？**

- 城市尺度小（古城+下关+海东三个核心区直径不超过 15km）
- 社群高度分层但**互相独立**：数字游民圈 / 客栈业主圈 / 本地白族圈 / 康养老人圈 / 宠物医院圈 互不重叠 ← **这正好是"社交圈不覆盖"假设最成立的场景**
- 宠物医院和救助站可数（10 家以内全部走访可行）
- 客栈、咖啡馆是天然的"信息张贴点"
- 数字游民群体技术接受度高 → 早期用户

**冷启动数据来源（已决定）：**

不爬取微博/小红书。**也不做平台代发**（详见第 8 条 + 不做清单）。改为：

1. 开发者本人加入本地 10+ 宠物群
2. 看到丢失/招领信息时**引导当事人自己来发**（不代发 · 让 ta 当首批种子用户）
3. 在合作宠物医院/客栈/救助站张贴小程序码
4. 长期方向：合作机构入驻（救助站/宠物医院开自己的实名账号，自己发自己维护）

**冷启动期 0-2 周可能只有 5-15 条数据**——这是默认假设。
**宁可空 feed，也不要假信任**（详见第 8 条）。所有设计（地图 tab 不做、信息流默认时间排序）都基于这个数据稀疏 + 真实优先的假设。

### 6. 教育只说一次 · 不抢夺注意力

凡是「告知 / 介绍 / 教育 / 引导」性质的元素——**用户认知一次即可**。第二次出现就是冗余信息，是对用户注意力的侵占。

**行业默认**：抢夺注意力（更醒目 / 更长时间 / 更频繁触达）→ 用户疲劳
**我们反其道**：克制 + 信任用户记忆 → 用户尊重感

具体落地：

| 元素 | 行为 |
|------|------|
| 启动隐私 modal | 同意一次后不再出现 |
| 公众号引导 modal | 关掉一次后不再主动弹（菜单里仍可找到入口） |
| 首页 feed-stat-bar（自我介绍条）| 进入 5s 后自动收起 · 用户滚动立即收起 · 永久不再显示（localStorage） |
| 发布表单顶部「关于悬赏」说明 | 仅首次进入发布表单时显示 |

**判断标准**：「这个元素出现第二次有新增价值吗？」如果答案是「没有」，就让它消失。

#### 第 6 条延伸 · 解释什么，就唤起什么

用户没意识到的问题，**不要主动"解释清楚"——你的解释本身就是提醒**。

| ❌ 反例（凭空唤起疑虑）| ✅ 正确做法（设计上消解，文案上不提）|
|---|---|
| "你的手机号永远不会公开" | 默认就不收集手机号 → 文案完全不提手机号 |
| "这里没有等级，每个人都平等" | 默认就没有等级 → 不需要任何文案 |
| "我们不刷数据" | 默认就真实 → 不需要"声明真实" |
| "参考话术 · 点击复制" | 默认相信用户会说话 → 不给脚本（v10 r13 砍）|

**核心原则**：「设计上不需要解决的问题，文案上一次都别提」。
**深层逻辑**：每条"安抚性解释"都在**反向唤起用户的疑虑**——本来用户没在想这件事，看完文案反而开始想了。

### 7. 不为焦虑的人增加心理负担

用户打开这个产品时本就**处在情绪低谷**——丢宠物的失主、捡到不知所措的好心人。
**UI 不应叠加心理负担**。

具体落地：

| 反例（看似合理实则有害）| 替代（情感安全）|
|------|------|
| 「⚡ 黄金时段剩 5h50m」倒计时 | 「10 min 前」中性时间 + 「附近有 N 人在帮」希望叙事 |
| 「关键期 · 找回率 30%」概率展示 | 不显示概率（数据正确但人性错误）|
| 醒目的红色「急」徽章满屏闪 | 平等的卡片视觉，让 feed 平静 |
| 「您已 3 天未发布新进展」催促 | 不催促，失主自己决定 |

**判断标准**：「这个元素让失主**更焦虑**还是**更安心**？」
**深层原理**：损失框架（loss frame，"剩余 X 小时"）触发损失厌恶心理 → 焦虑感 > 希望感。
**急诊医学的「黄金时段」是给医护人员看的，不是给患者家属看的**——不要搬错对象。

### 8. 真实 > 看起来繁荣

宁可 feed 是空的，也不要**虚假的活跃感**。每一条假数据 / 代发数据 / 刷的浏览量都是**长期信任债**。

| 行业默认 | 我们反其道 |
|------|------|
| 冷启动期刷"看起来繁荣"的数据 | 承认我们刚起步，每条都是真的 |
| 代发其他平台/微信群信息（标"已授权"）| **不做代发**——授权管理法务级 / 数据时效失主不回来更新 / 个保法红线 |
| 假浏览量 / 假评论 | 真实数字（即使个位数）|
| 用"我们汇聚了 X 家机构"包装 | 数据条副标题：「真实失主自助发布 · 拒绝悬赏 / 转发」 |

**关键认知**：用户的"真实+情感连接"是产品最终的护城河。一次"假繁荣"被发现，比 feed 显空伤害大 10 倍。

冷启动的真实路径：**让信息发布的"权属"始终归到具体人**（失主自己 / 合作机构自己），平台不做"代理人"。

---

## 核心哲学的精神归一

上面 8 条本质上指向同一件事：

> **失主已经焦虑，捡到的好心人已经在帮忙——我们能做的最有用的事，是不打扰、不评判、不假装。**

这个产品的成功不是「用户停留时长」，是「**找回的宠物数量**」。前者多了反而是失败信号——意味着用户在我们这里耗着没解决问题。

## 奥卡姆案例库（反面教材 · 看似巧妙实为债务）

每个曾经"看起来很好的功能"都被砍掉了。记录在此防止后人重蹈覆辙。

| 案例 | 看起来的价值 | 真实代价 | 砍的理由 |
|------|------|------|------|
| **善行值积分系统** | 行为激励 / 留存抓手 | 冷启动期"空跑"无意义；用户量 < 1000 时排行榜无意义、积分换不出去 | 用「轻量信任」（行为数统计）替代 |
| **临时照料 C2C** | 多一个使用场景 | 疾病传播 / 二次走失 / 意外伤亡 / 变相占有 / 法律灰色 | 风险 >> 收益；V2 跟救助站合作以"转交救助站"重启 |
| **「📌 已授权代发」标签** | 解决空 feed（冷启动巧妙方案）| 授权管理工作量法务级 / 失主不回来更新数据时效失控 / 个保法风险 / 数据所有权不清 / 永远扫不完群 | 看起来巧妙，实为**长期债务** |
| **⚡ 紧迫度倒计时（phase）** | 视觉差异化 + 行业共识（动物行为学）| 「剩余时间」是损失框架 → 触发失主焦虑而非给希望；"黄金/关键/长期"是平台对失主处境的评判，是俯视姿态 | 数据正确但人性错误；急诊医学概念误用 |
| **`👀 浏览量`** | 标准社交媒体指标 | 浏览量不影响"该不该帮"；寻宠不是新闻火不火无意义；容易被刷 | 删除，仅保留「🤝 N 接力」（真正"在帮忙的人数"）|
| **「附近的小朋友 · 14」section header** | 标记 feed section + 显示数量 | 整个 feed 就是"附近的小朋友"，自说自话；数据条已表达数量；"查看全部"在 feed 全列时无意义 | 删除，把"查看全部"功能整合到数据条右侧汇总徽章 |
| **「🔒 私有特征」字段** | 反冒领工具 / 失主"防御感"安全锚 | 真实使用场景占比 < 5%（80% 是目击线索不需要核对；15% 见面时宠物反应+疫苗本才是真核对）；**严重误伤真实好心人**（好心人不会刻意记宠物独家特征，被考时答不上 → 失主误判为骗子）| 真正的防御靠宠物反应+疫苗本+24h冷却；失主的"防御感"由 `contact-guide` 的核对指南承载，比一个误伤好心人的字段更安全 |
| **「发布者信息卡」(poster-card)** | 信任建立 / 实名认证背书 / 举报入口承载 | **典型信任剧场**：用户决策不依赖发布者信息（看见走丢的猫谁都会想帮）；「✓ 实名」**误伤未认证的真实失主**（懒得给身份证不等于骗子）；「JOINED 8 days · 已发布 1 条」是无信号统计（失主多数只发 1 次永远是 1）；"已发布 N 条"是用户行为画像 → 个保法红线 | 用户关注的是**宠物**，不是发布者；举报入口移到右上角 ⚠ icon（满足合规即可） |
| **「⭐ 守护者」等级徽章** | 行为激励 / 荣誉感 / 信任分层 | **等级化角色** = 平台对帮助者打分；和"实名认证"同病：未达守护者门槛的真实好心人被暗示"不够好"；与"每个帮助者平等"的哲学根本冲突；冷启动期没有数据基础，徽章授予标准模糊 | 用「已成功归还 N 次」「收到 N 封感谢」等**具体行为统计**替代，**不做等级**——让每条贡献本身被看见，而不是"达到 X 分就升级"|
| **「🗺 查看」假按钮**（dist-bar-map-btn）| 提示用户可以点击查看地图 | 整个 detail-dist-bar 已经是 clickable（cursor: pointer + hover），右侧再加一个 chip 是**冗余视觉装饰**；用户反馈"莫名其妙的悬浮按钮"——**说明真实用户被困惑了**（"这是按钮还是装饰？"）| 删 chip · 整个 bar 仍可点击 |
| **「🥺 我丢了宠物」卖惨表情** | 增加情感共鸣 / 拉近距离 | 🥺 是 emoji 语义里的「**可怜兮兮 / 卖惨求人**」；**失主自己**点开发布表单看到 🥺 → 暗示"你很可怜"→ **加深焦虑**；同理 🤗 包装"我捡到了"过度积极，但很多发现者其实是"无奈+不知所措" | 改 🔍（功能性 · 在找）/ 🏠（功能性 · 有个家收留），完全去情绪化 |
| **顶部"草稿"+ 底部"保存草稿"两个入口** | 多入口便利 | 一模一样的功能放两个地方 → 用户困惑「点哪个」；违背奥卡姆 | 删顶部，保留底部（用户填完表自然到底部）|
| **「🚪 注销账号」错误术语** | 给用户"销户离开"的入口（沿用 App 时代术语）| **微信小程序无账号概念**——用户身份基于 OpenID 自动建立，没有传统"注册-登录-注销"流程；「注销账号」是把 App 时代的术语硬塞过来，**用户看了不知道实际删什么**；同时"销户"暗示"你要离开"是负面框架 | 改名「🗑 清除我的数据」+ 列明清除范围（位置 / 聊天 / 帖子）+ 标明 30 天合规冷却期。**这是术语审视奥卡姆**：换平台时旧概念可能完全不适用 |
| **「联系方式」radio +「虚拟号中转拨打」选项** | 给失主隐私选择权 / 隐藏真实电话 | **多重病**：① **让焦虑用户做技术决策**（"虚拟号是啥？平台代接通是啥？0.3 元/分钟谁付？"）违背第 7 条；② **99% 用户默认接受推荐**，是个伪选择；③ **虚拟号是 MVP 不该做的重型基础设施**（电信合作 / 号码池 / 支付通道 / 计费 / 客服）；④ 电话联系不是真需求（拍照+留言比慌乱中打电话更省力）；⑤ **主动提"手机号"会让本不焦虑的用户开始担心隐私** | 砍整个 radio + selectContact JS + radio CSS；改为绿色「💚 通知保障卡」一行文案：「有人留线索时，立刻在小程序 / 微信内通知你」。MVP 不收集手机号、不做虚拟号；virtual_phone 字段从数据模型移除 |
| **`contact-guide`「参考话术 · 点击复制」3 条模板** | 帮路人/失主"会说话" · 降低沟通门槛 | **多重病**：① **暗含"你不会说话"的焦虑暗示**——提供脚本 = 暗示用户需要别人帮 ta 说话；② **第一句"方便描述特别之处"与 v10 r6 哲学冲突**（"考特征"思路已砍）；③ **「📋 复制」按钮制造尴尬感**——对方一看就知道是模板，真诚度立刻下降；④ **过度公文化**——真实场景里一句"我捡到一只白色比熊你来看看是不是你家的"比 3 句正式模板有效；⑤ 真心想沟通的人不需要模板，不想沟通的人模板也救不了 | 完全删（3 个 template HTML + copy 按钮 + .template/.copy-btn CSS）。**不加替代提示卡**——真诚地相信用户会说话。这是「解释 = 唤起」原则的最纯粹应用 |
| **「等待中存在感」捏造的瞬间叙事** | 失主等待线索时给"被支持感"，缓解焦虑 | **AI 协作时容易犯的错**：被"温暖体验"语言带跑，把好的形式（瞬间叙事）套到**无法追踪的伪数据**上。例如："古城北门的白桃咖啡馆看到了，转给店里 3 位常客" —— 微信明确**禁止追踪转发后的传播链**（隐私保护），「白桃咖啡馆」这种身份平台也无法识别（OpenID 全匿名）。**这违反第 8 条「真实 > 看起来繁荣」**，且更严重：用户一旦发现是假的，平台信任体系崩盘 | **完全不做**——接受"失主等待焦虑"是真实的事实，不用伪叙事稀释。把资源放在让真线索更快出现的真功能上（转发卡双码 / nearby-helpers / 公众号推送）。**判断标准**：每个数字/具体描述都要能回答「这数据真实从哪来？」答不出来 = 别写 |
| **「💡 关于核对」前置摘要卡**（contact-guide）| 在详细步骤前给个概括 / 安抚 | **三重病**：① 内容与下方 step-block 4 步**完全重复**（"宠物对失主反应、疫苗本、旧照片"和 step 1+2 一模一样）；② 文案含「考特征」「不需要让对方先描述细节」等**反向唤起**词（违反第 6 条延伸「解释 = 唤起」原则）；③ **视角错位**——给发现者看，但内容讲的是失主视角的"核对"，造成困惑 | 完全删整节 + CSS · contact-guide 直接给"地点→步骤→异常"3 段实操，不需要"前置摘要" |

**共同教训**：每个反例都"听起来合理"，但深挖到**工程成本 / 法律风险 / 情感影响 / 运营负担** 这些维度后，就会发现砍掉才是正确选择。

**判断模板**：评估一个新功能时，问 5 个问题：

1. **工程**：这个功能的长期维护成本是多少？数据怎么更新 / 同步 / 清理？
2. **法律**：涉及个保法 / 微信合规 / 金融监管的哪些红线？最坏情况是什么？
3. **情感**：会让失主更焦虑还是更安心？给希望还是制造压力？
4. **真实**：这个功能让数据更真实还是更"看起来繁荣"？
5. **运营**：上线后谁来持续维护？需要多少人力？这个角色可持续吗？

任何一项答案不理想，**默认不做**。

## 关键设计决策的演进轨迹（v1 → v10）

| 版本 | 关键决定 | 哲学转向 |
|------|---------|----------|
| v1-v2 | 强制走完线上认领核对流程 | 默认假设：平台必须留痕 |
| v3 | 认领核对改为"沟通建议 + 自检清单" | 开始接受：核对发生在线下，平台只是辅助 |
| v4 | 砍掉"在平台对话内沟通"的强调，话术模板只留"复制"按钮 | 翻转：**不在乎用户在哪聊**，不强制留痕 |
| v6 | 归还流程从 4 屏砍到 3 屏；自检 checklist 砍成静态参考；一键归还 | 落地：平台不挡用户的路，配合用户的真实心智（"我只想标记一下，拿善行值"） |
| v8 r1 | 砍善行值积分体系，改「轻量信任」（行为数统计） | 哲学：冷启动期数值是空跑，**用真实行为统计建立信任更直接** |
| v8 r2 | 砍发布字段 3 项 / Quick Reply 6→3 / 个人中心菜单 13→10 | 奥卡姆剃刀：**砍掉伪需求，体验反而更好** |
| v8 r3 | 砍「临时照料」整条功能线（chip / 字段 / 状态 / 入口） | 风险面广 / 需求弱 / 非核心闭环；**敢砍最像"功能"的功能，因为它不是 MVP 的命** |
| v10 r1 | 团队合并：v8 UI + 队友地图模块 → v10 唯一主原型；同步砍照料 + 死链清理 + 转发卡双码 + 冷启动三件套 | 团队协作：**两个分叉收敛为一个 source of truth** |
| v10 r2 | 砍「📌 已授权代发」标签系统（CSS / 字段 / 渲染逻辑全删）| **真实 > 看起来繁荣**：宁可 feed 空，也不要假信任。代发是"看起来巧妙实为长期债务"的典型 |
| v10 r3 | 砍紧迫度系统（phase / ⚡ / 倒计时 / urgent 视觉强化）| **不为焦虑的人增加心理负担**：「剩余时间」是损失框架，触发焦虑而非给希望；急诊医学概念用错对象 |
| v10 r4 | nearby-helpers 升级为"希望叙事"主力（≤ 5km 都显示 · 三档文案 · 薄荷绿）| 用「附近有同行者」的社区感取代倒计时压力，情绪锚定在被支持 |
| v10 r5 | 数据条副标题改为产品价值观叙事；自我介绍条 5s 自隐 | **教育只说一次**：用户认知一次即可，不强加重复信息 |
| v10 r6 | 砍「🔒 私有特征」字段（表单 + 详情页 + 联系指南 + JS + CSS 全删）| 真实使用 < 5%；**严重误伤真实好心人**；防御感由 contact-guide 核对指南承载更安全 |
| v10 r7 | 砍「发布者信息卡」(poster-card)，举报入口移到右上角 ⚠ icon | 信任剧场：用户决策不依赖发布者；「实名」误伤未认证真实失主；「已发布 N 条」是个保法红线（用户行为画像）|
| v10 r8 | 砍「⭐ 守护者」等级徽章；删 detail-dist-bar 右侧「🗺 查看」冗余假按钮 | 等级化 = 打分 → 与"帮助者平等"哲学冲突；用具体行为统计替代。"假按钮"用户反馈"莫名其妙"= 真实困惑 |
| v10 r9 | 发布页 4 项优化：① 删顶部"草稿"sub-action（与底部重复）② seg 表情 🥺/🤗 → 🔍/🏠（去卖惨/过度积极）③ breed-chips 6→5+「其他品种」自由输入 ④ compliance-note 升级为价值观腔（"邻里互助网络，不撮合金钱交易"）| 信息发布页第一性原理审视：表情卖惨让失主焦虑；底部+顶部双草稿入口困惑；品种 chip 周到性兼顾（5 常用+其他） |
| v10 r10 | （队友）5 项人性化功能：温情横幅 / 看到了按钮 / 关注按钮 / 失主发进展 / 归还统计 | 两个张力点先保留观察（有队友巧思）|
| v10 r11 | 砍「🚪 注销账号」错误术语 → 改「🗑 清除我的数据」+ 列明范围 + 30 天冷却 | **平台术语审视**：微信小程序无账号概念，沿用 App 时代术语会让用户困惑；同时"销户"是负面框架。合规要求保留（个保法 + 微信规范），但要正确表达 |
| v10 r12 | 砍发布页「联系方式」radio + 虚拟号选项；改为「💚 通知保障卡」一行；MVP 不收集手机号；privacy modal 4 项收集 → 3 项；「清除我的数据」文案同步去掉"手机号" | 让焦虑用户做技术决策 / 99% 不切换 / 虚拟号重型基础设施 / 不需要主动提"手机号"让用户担心隐私 |
| v10 r13 | 砍 `contact-guide`「参考话术 · 点击复制」3 条模板 + .template/.copy-btn CSS；**不加替代提示卡**——真诚相信用户会说话 | 提供脚本 = 暗示"你不会说话"；「📋 复制」降低真诚度；过度公文化。**沉淀新原则**：第 6 条延伸「解释什么，就唤起什么」——设计上不需要解决的问题，文案上一次都别提 |
| v10 r14 | 升级 detail-lost 已有线索区 · 从"日志列表"改为「**真实瞬间 + 行动启动器**」：📍地点·时间 + 第一人称引语 + [🗺 现在去看 / 💬 问发布者] 双按钮 | **失主收到线索后真实下一秒是去现场询问/找监控**——线索不该是信息条目，应该是行动起点。同时把"等待中存在感"反馈方案否决（避免捏造无法追踪的数据）。**沉淀防呆**：CLAUDE.md 加新条款"不要捏造无法追踪的数据" |
| v10 r15 | 砍 contact-guide「💡 关于核对」前置摘要卡 + .info-soft CSS · step-block 标题「见面时建议这样核对」→「见面时这样做更稳妥」| 前置摘要内容与下方 4 步 step-block 100% 重复 + 文案含"考特征"等反向唤起词；"核对"在发现者视角下隐含"我被审查"压力，改为日常的"更稳妥" |

**整个演进的精神是：从"产品做尽职调查"转到"产品配合用户完成已经发生的事"**。

中间精神补充：**从"功能丰富"转到"功能克制"**——MVP 应该是用户走得通的最短路径，不是设计稿能堆的所有可能。

最近的精神补充：**从"产品中心"转到"用户情感中心"**——不抢夺注意力、不制造焦虑、不假装繁荣，把所有"中心舞台"让给用户的真实需求。

## 不做清单（重要：避免重复讨论）

M0 阶段以下功能**明确决定不做**，下次有人提起请直接 reference 这里：

| 功能 | 不做原因 |
|------|---------|
| 独立的地图 tab（看 pin 浏览） | 冷启动数据稀疏，地图反而显空；列表 + 距离标签已足够 |
| 邀请奖励善行值 | 微信审核风险（拉新返利红线） |
| 爬取微博/小红书冷启动数据 | 法律 + 声誉 + 审核风险三重叠加 |
| 平台担保的悬赏 / 感谢金 | 平台不介入金钱交易 |
| 善行值兑换 / 捐赠 | M0 不做；V2 探索 |
| 完整的志愿者后台 SOP | 用户 < 200 时开发者+合作者自己审核足够 |
| 一键生成海报 | 截图分享已够用；模板设计成本高 |
| 跨城市切换 | 大理跑通再说 |
| 完整的反作弊后台（抽检/关联监控） | 用户量起来再做 |
| 复杂排序算法（混合距离+时间+互动） | M0 用纯时间排序，等数据起来再调 |
| **临时照料（C2C 私人代管）** | 疾病传播 / 二次走失 / 法律灰色 / 需求弱；V2 跟救助站合作后以「转交救助站」形式重启 |
| **「📌 已授权代发」标签系统** | 授权管理工作量法务级 / 数据时效失主不回来更新 / 个保法风险 / 数据所有权不清 / 永远扫不完群——长期债务大于冷启动收益。详见第 8 条 + 奥卡姆案例库 |
| **⚡ 紧迫度倒计时（phase badge）** | 「剩余时间」是损失框架→触发失主焦虑而非给希望；"黄金/关键/长期"是平台对失主处境的评判，俯视姿态；急诊医学概念用错对象。详见第 7 条 + 奥卡姆案例库 |
| **`👀 浏览量` 指标** | 浏览量不影响"该不该帮"；寻宠不是新闻火不火无意义；容易被刷。仅保留「🤝 N 接力」 |
| **悬赏 / 感谢金平台担保** | 微信合规红线 + 灰色产业链（赏金猎人 / 拒付 / 勒索）；我们的差异化定位 = **「互助网络」而非「悬赏寻宠」**——主动筛选用户群 |
| **"急" 视觉强化（深阴影 / 高光线 / 大字号）** | 每张卡视觉平等，不强化任一类卡。强差异化 = 给失主心理压力 |
| **「🔒 私有特征」字段** | 真实使用 < 5%；严重误伤真实好心人；失主的防御感由 `contact-guide` 核对指南承载即可。详见第 2 条 + 奥卡姆案例库 |
| **「发布者信息卡」(poster-card)** | 用户关注的是宠物不是发布者；「✓ 实名」误伤未认证的真实失主；「已发布 N 条」是用户行为画像 → 个保法风险；举报入口已移到右上角 ⚠ icon。详见第 8 条 + 奥卡姆案例库 |
| **「⭐ 守护者」等级徽章** | 等级化 = 平台对帮助者打分；未达门槛的真实好心人被暗示"不够好"；与"每个帮助者平等"哲学根本冲突。用具体行为统计（已归还 N 次 / 收到 N 封感谢）替代等级。详见第 8 条 + 奥卡姆案例库 |
| **冗余的"假按钮"装饰**（如 dist-bar-map-btn）| 整体 clickable 的容器内再加 chip 提示是冗余；用户反馈"莫名其妙"= 真的造成困惑。原则：**整体可点击就不要装饰按钮** |

## 复用这份文档的方式

- **写新功能时**：用「奥卡姆案例库」末尾的 **5 个判断问题**（工程/法律/情感/真实/运营）评估。任何一项答案不理想 → 默认不做
- **设计新 UI 元素时**：先问"这符合 8 条哲学吗？"特别检查第 6/7/8 条（教育只一次 / 不增加焦虑 / 真实优先）
- **被合作者/路人问"为什么不做 X"时**：直接 reference「不做清单」或「奥卡姆案例库」
- **每次原型迭代后**：把新决策写到"决策演进轨迹"表
- **每次砍掉一个功能后**：如果是"看起来巧妙实为债务"的典型，写入「奥卡姆案例库」防止后人重蹈
- **每次重大讨论后**：如果产生了新哲学，更新这里；如果只是局部决策，写到对应的 `0X-*.md`
