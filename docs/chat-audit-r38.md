## 二、不符合产品设计逻辑

### 2.1 「🎉 已找到」chip 与右上角菜单「我把 ta 交给主人了」**职责严重重叠**

```
chip 「🎉 已找到」 → 只发一句聊天消息 · 平台不知道
菜单 「我把 ta 交给主人了」 → 平台层标记 + 24h cooldown + 帖子关闭 + 善行值
```

用户点 chip 期望"事件结束"，结果只是发了句话。**机制和行为脱节** —— 反例库 #12「双入口冗余」典型。

**建议**：删除 chip · 保留菜单作为唯一入口。

### 2.2 mock 对话内容没考虑视角

`chat-msg me` 写死「我家果儿脖子右侧白色心形斑点」：
- view-as-owner（失主）说这话 → 合理 ✓
- view-as-finder（拾主）说这话 → 不合理 ✗（拾主不会说"我家的"）

**影响**：不影响真实产品（用户自己输入），但**演示 prototype 时切换视角会露馅** · 老板/客户/合作伙伴看到时会被质疑。

**建议**：mock 对话按 view-as 切换内容 · 或显式标注"以下为失主视角对话演示"。

### 2.3 入口路径漏洞 —— 可能藏类似 r29 的 wrapper bug

我们 r29 修过的 `showScreen` wrapper 吞参 bug 同性质问题可能存在于 chat 屏：

```js
// detail 屏线索卡「💬 问陈*」入口
<div class="clue-action" onclick="showScreen('chat')">💬 问陈*</div>
```

直接 `showScreen('chat')` **不调用 `openChatAs()`** → view-as 状态保留上次值 → quick-replies 可能错。

**建议**：把所有进 chat 屏的入口都规范为 `openChatAs(role)`。需要 grep 全文确认所有入口。

## 三、没做完全（实施时需补齐）

### 3.1 撤销归还的"通知对方"机制缺失

`undoReturn()` 只删除自己 UI 上的 cooldown-bar 和 system-msg，**对方屏上的 cooldown 不会消失**，也没有「对方撤销了归还标记」的系统消息。

**实施时必须补齐**：socket/推送链路 · 服务端写入撤销事件 · 双方屏同步状态。

### 3.2 mock 对话固化在「客栈老板娘」一个场景

不论从哪个入口进 chat（💬 私聊失主 / 💬 私聊发现者 / 消息列表 / 线索卡的 💬 问 xx）都看到同一份对话。

**影响**：演示时无法体现"按场景切换"。

**建议**：至少给 prototype 加 2-3 个不同 mock 对话（失主-拾主 / 拾主-失主 / 邻居-失主）按入口切换。

### 3.3 「📍 发送位置」按钮没有合规边界

```js
onclick="alert('📍 发送位置\n（小程序会调用 wx.chooseLocation）')"
```

精确位置共享是隐私敏感操作 · 地图屏已模糊到 500m · 但 chat 这里直接共享精确点。

**建议**：
- MVP 加确认环节「将分享精确位置给对方 · 确认？」
- 或 V2 才上线 · MVP 只支持文字 + 照片

## 四、过度设计 / 没必要做

### 4.1 quick-replies 13 个 chip 太多 · 奥卡姆违背

```
finder: 📍 我看到了, 🏠 我收着 ta
owner: 📍 马上过来, 📹 视频确认
通用: 📷 发张照片, 🐾 描述特征, 🤝 约见面, 🎉 已找到, ✅ 好的, 🙏 谢谢
```

13 个 chip 横向滚动 → 认知负担大 · 真实使用率低。

**建议保留 5-6 个核心 chips**：
- 删「🎉 已找到」（和菜单重叠 · 见 #2.1）
- 删「✅ 好的」（太通用 · 打字更快）
- 删「🙏 谢谢」（同上）
- 删「📹 视频确认」（小程序 video 调用麻烦 · 实际很少用）

### 4.2 chat-menu 双举报入口 · 反例库 #12

```
⚠ 举报这条对话
🚫 举报 / 拉黑用户
```

用户困惑"点哪个" · 建议合并为一个「⚠ 举报对方」入口 · 进 sheet 后再细分。

## 五、团队产品化思维 4 条普适教训

这次审视提炼出的洞察，可写进文档或作组内分享：

### 5.1 入口路径漏洞律（来自 r29 经验）

> 每给核心函数（`showScreen`、`openChatAs` 等）加新参数，**必须 grep 所有调用点 + wrapper**，否则会被默默吃掉。

**真实事故**：r19~r28 反复修视角推导 bug，实际从未生效。根因是 r10 时代的 wrapper 只声明 1 个参数，吞掉了 r19 之后加的 `role` 参数。

### 5.2 mock 必须反映视角

> 写 mock 内容时如果产品有多视角，mock 必须按视角动态化，否则演示露馅。

适用：chat 对话内容、详情页文案、my-posts 列表、空状态提示。

### 5.3 职责重叠是奥卡姆杀手

> 两个按钮做"几乎一样的事"几乎总是有一个该删，或者一个该升级一个该降级。

判断方法：用户点 A 和点 B 的**后果不同**吗？
- 后果相同 → 删一个
- 后果不同（如 chip vs 菜单）→ **必须让用户能区分** · 不然机制和行为脱节

### 5.4 撤销机制要双向

> 任何"标记 + 撤销"机制必须考虑对方的同步状态，否则会出现"我撤销了但对方屏上还在 cooldown"。

适用范围：归还闭环、消息撤回、争议撤销、订单状态等所有双方都能看到的状态。
