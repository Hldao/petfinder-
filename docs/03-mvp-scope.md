## 一、MVP 整体规划

### 目标场景

大理本地的寻宠 / 拾遇互助：

```
失主发布 → 路人 / 拾主看到 → 留线索 / 联系 → 沟通确认 → 归还成功 → 写感谢信
```

|---|------|------|------|
| 1 | 信息流（首页 + 搜索）| feed / search | ✅ |
| 2 | 详情页（寻宠 + 招领）| detail-lost / detail-found | ✅ |
| 3 | 发布表单（双场景）| publish | ✅ |
| 4 | 站内聊天 | messages / chat | ✅ |
| 5 | 公众号关联 | mp-modal | ✅ |
| 6 | 启动隐私协议 | privacy-modal | ✅ |

### V2 再做（不在 MVP 内）

- ❌ **善行值积分系统**（线索 +2 / 归还 +6 等数值规则）← **MVP 阶段砍掉**
- ❌ 善行值流水页 / 徽章详情页 / 排行榜
- ❌ 多级守护者等级（普通 / 守护者 / 大师 / ...）
- ❌ 积分换公益捐赠
- ❌ **临时照料（C2C 私人代管）** ← **MVP 阶段砍掉**（风险面广 / 需求弱 / 非核心闭环）
- ❌ 转交救助站（V2 跟救助组织谈合作后以专业机构托管形式重启）
- ❌ 视频通话 / 语音通话（需资质）
- ❌ 真正的群聊（合规复杂）
- ❌ 跨城市（先打透大理）
- ❌ AI 智能匹配（需训练数据）
- ❌ 感谢信合集 / 月报

### 关于善行值的取舍（重要决策）

冷启动期（用户 < 1000）积分系统是「空跑的」—— 没有用户基数时，
排行榜无意义、徽章威信不够、积分用户拿到也用不出去（合规又禁止兑换）。

MVP 采用「**轻量信任**」替代「重量积分」：

| MVP 保留 | MVP 砍掉 |
|---|---|
| 实名认证 ✓ | 善行值数值（"善行值 48"）|
| 「已成功归还 N 次」具体行为统计 | 守护者徽章 / 多级等级（v10 r8 砍 · 详见 06 第 8 条 + 奥卡姆案例库）|
| 「收到 N 封感谢信」 | 积分流水页（points-history 整屏）|
| 「提供 N 条有效线索」 | 复杂的积分发放规则 |
| | 徽章墙 |

**用具体行为统计替代抽象数字**，建立信任反而更直接。
触发条件达成（用户量 ≥ 5000 / 行为数据丰富）后再启用 V2 积分。

### 关于「临时照料」的取舍（重要决策）

MVP 阶段**砍掉「临时照料」**整条功能线（筛选 chip / 发布字段 / 详情区块 / 个人中心入口）。

**为什么砍：**

| 维度 | 判断 |
|------|------|
| **风险面** | 疾病传播（猫瘟/犬瘟传染照料者宠物）/ 二次走失 / 意外伤亡 / 变相占有 / 法律灰色（私人无委托保管）|
| **需求强度** | 能提供照料的用户占比极低（需笼舍、时间、宠物经验）；多数捡到者只想"快速交接"|
| **闭环必要性** | 不是「信息匹配→沟通→归还」核心闭环的必经环节 |
| **替代路径** | 真有人愿意照料，在聊天中自然沟通即可，不需要产品层面强引导 |

**砍掉影响：**

- 数据模型 `posts.type` 从三态 `lost/found/care` 简化为两态 `lost/found`
- 个人中心移除「我可以临时照料」开关
- FEED_DATA 中原 3 条 care 数据归并为 found
- 善行值规则同步移除「临时照料 +5」（见 `docs/01-points-system.md`）

**V2 重启策略：** 等跟救助组织谈好合作，以「**转交救助站**」（专业机构托管，平台仅做交接撮合）形式重新引入，规避私人代管的法律/疾病/纠纷风险。

### 永远不做（合规红线）

- 🚫 充值 / 提现 / 积分商城
- 🚫 悬赏金 / 感谢金
- 🚫 拉新得积分（分销）
- 🚫 引导加微信 / 留电话

## 二、模块详细规格

### 模块 1 · 信息流（首页 + 搜索）

**屏幕**：`feed` + `search`

#### 1.1 首页 feed 功能清单

| 功能 | 描述 |
|------|------|
| 顶部导航 | 城市标签（**固定大理**，无下拉）/ 搜索栏 / 通知按钮 |
| 筛选 chip（3 选 1）| 全部 / 寻宠 / 招领 |
| 排序 Action Sheet | 最新发布 / 距离最近 |
| 卡片列表 | 单排 + 数据驱动渲染（FEED_DATA → 14 张 mock）|
| 卡片底部统一规则 | `💬 留言数 · 👀 浏览量 · ⤴ 转发` |
| 转发 Modal | 准朋友圈分享卡（含品牌 / 大图 / 信息 / mock 二维码）|
| 中央凸起 FAB | 4 个 tab 屏共用，点击进发布 |
| 底部 Tab | 首页 / 地图 / 消息 / 我的 |

#### 1.2 搜索屏 search 功能清单

| 功能 | 描述 |
|------|------|
| 输入框 | 真实输入 + 自动聚焦 + 清除按钮 |
| 默认状态 | 最近搜索（mock）+ 大理热门搜索（8 个本地地标 / 品种）|
| 输入中 | 实时搜索建议（关键词高亮 `<mark>`）|
| 结果状态 | 复用卡片样式 + 匹配数量 |
| 空状态 | 友好提示 + 引导发布 |

#### 1.3 数据模型

```
posts (帖子)
├── id
├── type: 'lost' | 'found'
├── status: 'pending_review' | 'approved' | 'returned' | 'expired'
├── pet: { name, breed, gender, age, photos[] }
├── location: { city, district, address, lat, lng, geohash }
├── time: timestamp
├── description (公开)
├── contact_method: 'im' | 'virtual_phone'
├── poster_id → users
├── stats: { views, comments, shares, clues }
└── created_at / updated_at
```

#### 1.4 接口约定（粗）

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/feed?city=&filter=&sort=&page=` | 卡片列表（分页）|
| GET | `/search?q=&city=` | 搜索 |
| POST | `/shares` | 记录分享行为（统计）|

#### 1.5 合规要点

- 仅 `status = approved` 的内容才出现在 feed
- 9 天前内容自动透明度降低（前端 `expired` class）
- 位置精度模糊到 500m（数据库存 geohash 加密）
- 转发卡片含 mock 二维码（真实环境用 `wx.showShareImageMenu`）

### 模块 3 · 发布表单

**屏幕**：`publish`

#### 3.1 模式切换

```
seg: [🥺 我丢了宠物] [🤗 我捡到了]
              ↓ 切换
共用字段 + 各自专属字段
```

#### 3.2 字段对照

| 字段 | 我丢了（lost）| 我捡到了（found）|
|------|---------------|------------------|
| 照片 | ✅ ≥ 1 张 | ✅ ≥ 1 张 |
| 宠物类型 | ✅ 单选 | ✅ 单选 |
| 名字 | ✅ | ❌（捡到的不知道）|
| 品种 / 外观 | ✅ + 联想 chips | ✅ + 联想 chips |
| 性别 / 年龄 | ✅ | ✅（估计值）|
| 地点（**走失** / **发现**）| ✅ Action Sheet 选 | ✅ Action Sheet 选 |
| 详细地址 | 选填 | 选填 |
| 时间（**走失** / **发现**）| ✅ Action Sheet 选 | ✅ Action Sheet 选 |
| **目前位置** | ❌ | ✅ |
| **健康状况** | ❌ | 选填 |
| 公开描述 | ✅ ≥ 10 字 | ✅ ≥ 10 字 |
| 联系方式 | ✅ radio | ✅ radio |

> ⚠ 原方案有「私有特征」字段（仅 lost 模式 + 4 个示例 chip），v10 r6 已砍。详见 `docs/06-product-philosophy.md` 第 2 条。

#### 3.3 联想数据

**品种 chips（按宠物类型联动 · 5 个常用 + 1 个「+ 其他品种」自由输入）**：

| 类型 | chips（5 常用） | 末尾 |
|------|-------|------|
| 猫 | 橘猫 / 田园猫 / 英短 / 美短 / 布偶 | + 其他品种 |
| 狗 | 中华田园犬 / 金毛 / 柯基 / 泰迪 / 比熊 | + 其他品种 |
| 其他 | 兔子 / 仓鼠 / 龙猫 | + 其他品种 |

「+ 其他品种」chip 用虚线边框区分（视觉暗示"自由输入"），点击 → 清空 input 并聚焦。

> v10 r9 决策：周到兼顾常见品种快速填充 + 长尾品种自由输入。详见 `docs/06-product-philosophy.md`。

**地点 chips（大理本地）**：

- 使用当前位置 / 古城北门 / 人民路三月街 / 下关泰安路 / 喜洲古镇 / 海东

**时间 chips**：

- 刚刚（5 分钟内）/ 1 小时前 / 今天上午 / 今天下午 / 昨天 / 自定义


#### 3.4 校验规则（前端）

| 规则 | 文案 |
|------|------|
| 照片 ≥ 1 张 | "请至少上传 1 张照片，帮助大家识别" |
| 地点必选 | "请选择地点" |
| 时间必选 | "请选择时间" |
| 描述 ≥ 10 字 | "描述至少 10 字，详细描述能帮助匹配成功率提高 3 倍" |
| 照片 ≤ 9 张 | "最多上传 9 张照片" |

#### 3.5 接口约定

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/posts` | 创建（先 pending_review）|
| POST | `/posts/:id/photos` | 照片上传 |
| GET | `/locations/recommended?city=` | 推荐地点 |

#### 3.6 合规要点

- **发布须知前置告知**（顶部红色卡片）
- 提交后调用 `security.msgSecCheck`（文字）+ `security.imgSecCheck`（图片）
- 未通过审核**不入库 / 不出现在 feed**
- 手机号字段加密（AES-256）存储
- 联系方式默认推荐"仅站内消息"

#### 4.1 messages 列表

| 来源 | 视觉区分 | 示例 |
|------|----------|------|
| 公众号通知 | 薄荷绿头像 + 「服务号」chip + 置顶 | "📍 古城北门附近有人捡到一只蓝猫" |
| 系统通知 | 灰色"系"字头像 | "陈* 的线索已被标记为有用" |
| 1v1 对话 | 宠物照头像 + 关于 chip | "好的我现在过来，您能视频先确认一下吗？" |

#### 4.2 chat 屏

| 区块 | 内容 |
|------|------|
| Header | ← 返回 / 对方昵称 / ⋯ 更多 |
| 宠物 card（顶部）| 缩略图 + 名字 + 状态 chip，点击跳详情 |
| 消息流 | 我的（橘色右对齐）/ 对方（白色左对齐，含阴影）/ 时间分隔 |

| **Quick Reply 6 个** | 横向滚动 chips |
| 输入栏 | 📷 / 📍 / input / ↑ |

#### 4.3 Quick Reply 模板

按"路人联系失主"视角排列：

| 模板 | 填入内容 | 视觉 |
|------|----------|------|
| 📍 我看到了 | "📍 我在 " (留位置占位) | 橘色 primary |
| 📸 我拍到了 | 完整句 | 灰底 |
| 🏠 我收着 ta | 完整句 + 见面询问 | **薄荷绿 action**（强意图）|
| 🕐 约时间 | 约见面句 | 灰底 |
| ❓ 视频确认 | 防冒领关键问题 | 灰底 |
| 🙏 谢谢 | 基础礼貌 | 灰底 |

#### 4.4 关键词过滤（前端预筛）

正则匹配命中即拒：

```
微信 | wechat | wx\s*[:：] | qq | 加我 | 私聊 | 联系方式 | 1[3-9]\d{9}
```

拒绝文案：「⚠ 消息含敏感内容（如微信号/电话/引导平台外联系）· 请使用站内消息沟通」

#### 4.5 数据模型

```
chats (会话)
├── id
├── post_id → posts
├── participants: [user_a, user_b]
├── created_at
└── last_message_at

messages (单条消息)
├── id
├── chat_id → chats
├── sender_id → users
├── content
├── type: 'text' | 'image' | 'location' | 'system'
├── status: 'sending' | 'sent' | 'blocked' | 'reviewing'
├── created_at
└── deleted_at
```

#### 4.6 接口约定

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/chats` | 我的会话列表 |
| GET | `/chats/:id/messages?before=&limit=` | 消息历史 |
| POST | `/chats/:id/messages` | 发送消息（同步调 msgSecCheck）|
| WebSocket / 轮询 | `/chats/:id/subscribe` | 实时更新 |

#### 4.7 合规要点

- **每条消息**必过 `msgSecCheck`（前端关键词过滤只是预筛）
- 图片消息必过 `imgSecCheck`
- 消息留存 ≥ 6 个月
- 注销账号时**保留消息**（脱敏处理）满足取证要求
- 进入 chat 时自动请求订阅消息授权（用于"对方新消息"推送）

### 模块 5 · 公众号关联

**目的**：解决推送 / 留存 / 传播痛点

#### 5.1 公众号定位

- **类型**：服务号（不是订阅号）
- **认证**：300 元/年（个体工商户主体已可申请）
- **关键能力**：
  - 关联小程序（自定义菜单跳小程序）
  - 服务通知模板（不受群发 4 次/月 限制）
  - 跟好友消息同级显示

#### 5.2 服务通知模板（待申请）

| 场景 | 模板内容示例 |
|------|--------------|
| 附近匹配 | "📍 古城北门附近有人捡到一只蓝猫，是你家胖丁吗？" |
| 新线索 | "📌 陈* 在洋人街看到了胖丁，留下了线索" |
| 归还成功 | "🎉 胖丁顺利回家了，谢谢你！→ 写感谢信" |
| 申诉处理 | "⚖ 你的申诉已处理 → 查看详情" |

#### 5.3 在小程序内的接入点

| 触点 | 时机 | 效果 |
|------|------|------|
| 启动 Modal | 用户首次同意隐私协议后自动弹 | 第一次进 app 建立公众号认知 |
| 个人中心菜单 | 「📣 关注公众号 · 推送更及时」| 主动想关注时找得到 |
| 消息列表置顶 | 公众号通知显示在列表顶部 | 让用户感受真实推送 |

#### 5.4 真实落地步骤

```
1. 注册公众号（1 周）
2. 服务号认证（300 元/年 + 1-3 天审核）
3. 关联小程序（公众号后台操作）
4. 设计自定义菜单（3 个一级：发布寻宠 / 看附近 / 我的）
5. 申请服务号模板消息（每个推送场景 1 个模板，1-2 周）
6. 持续运营：每周「大理回家了」周报 + 寻宠指南
```

### 模块 6 · 启动隐私协议

**屏幕**：`privacy-modal`

#### 6.1 4 个收集项明示

| 项 | 用途 | 处理方式 |
|----|------|----------|
| 📍 位置 | 发布走失/招领、附近匹配 | **精度模糊至 500m**（geohash 加密）|
| 📷 照片 | 宠物识别匹配 | **经平台审核** |
| 📞 手机号 | 联系（虚拟号转接）| **加密存储**，仅虚拟号供他人联系 |
| 💬 聊天内容 | 双方沟通 | **经审核送达，留存 6 个月** 满足监管 |

#### 6.2 流程

```
首次启动
  ↓
显示隐私 Modal（覆盖在首页上，不可绕过）
  ↓
用户点 [同意并继续]
  ↓
关闭隐私 Modal
  ↓ 0.25s 后自动
弹出 公众号关注 Modal（首次）
  ↓
用户可选择关注或稍后
```

#### 6.3 合规要点

- 必须在管理后台**填写「用户隐私保护指引」**
- 收集任何敏感信息前必须调 `requirePrivacyAuthorize`
- 提供「**注销账号**」入口（个人中心）
- 注销后**30 天宽限期**，期满彻底删除个人数据（聊天内容脱敏保留）
