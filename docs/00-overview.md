# 00 — 项目概览

## 定位

本地化宠物走失/招领信息互助平台，**基于"善意网络"理念**。

### 三条底线（不可破）

1. 平台不介入任何金钱交易（悬赏、感谢金等由用户自行线下）
2. 善行值仅作为荣誉记录，MVP 阶段不实现兑换或捐赠
3. 单城市启动，由开发者亲自运营 + 志愿者辅助

### 目标定位

- 阶段：**创业项目验证**
- MVP 阶段：完全不变现，验证需求与社区粘性
- V2 探索：抽成寻宠师服务 / 救助组织合作 / B 端会员（与"善意"定位保持兼容）

---

## 目标市场：云南大理

选大理而不是一线城市，原因：

- **城市尺度小**：古城 + 下关 + 海东三个核心区直径不到 15km，"3-15km 全城"地理参数都成立
- **社群高度分层、互相独立**：数字游民圈 / 客栈业主圈 / 本地白族圈 / 康养老人圈 / 宠物医院圈 等互不重叠——**正好是"社交圈不覆盖"假设最成立的场景**
- **冷启动成本可控**：宠物医院 + 救助站可数（10 家以内可全部走访），客栈/咖啡馆是天然张贴点
- **早期用户友好**：数字游民群体技术接受度高
- **运营深度可达**：开发者本人在地，可以亲自跑动建立合作

跑通大理后再考虑扩展。详见 `06-product-philosophy.md`。

## 团队与技术栈

| 项 | 配置 |
|---|---|
| 团队 | 开发者全职主导设计 + 代码（通过 Claude Code 实现）+ 1 名合作者全职 |
| 客户端 | 微信小程序**原生** wxml/wxss/js（不用 Taro/uni-app） |
| 后端 | 腾讯云开发 CloudBase（云函数 + 云数据库 + 云存储） |
| 地图 | 腾讯位置服务 |
| 推送 | 微信订阅消息 |
| 后台 | H5 + 云开发 web SDK，仅 2 个管理员账号 |
| 小程序类目 | **公益 / 动物救助**（不选"生活服务-社区"，审核标准更宽松） |
| 仓库 | https://github.com/Hldao/petfinder- |

四条不可妥协的架构原则（详见 `06-product-philosophy.md` § 3）：

1. 客户端**永远不直接写数据库**（反作弊的物理基础）
2. 善行值发放**集中**在单一云函数（避免多入口绕过校验）
3. 所有图片落库前必经**微信 imgSecCheck**（节省人工审核 80%）
4. M0 只用**一个云开发环境**（开发者 + 合作者直接部署）

## 核心业务流程

### 寻宠流程
```
失主发布 → 系统生成海报 → 他人提供线索 → 失主标记"有用" → 找回 → 双方确认归还 → 24h 冷却 → 善行值结算 → 失主写感谢信
```

### 招领流程
```
发现者发布 → 失主看到并联系 → 双方核对私有特征（视频/线下） → 线下交接 → 双方确认归还 → 24h 冷却 → 善行值结算
```

## 关键设计决策

### 1. 善行值激励重排（详见 01）

原方案"线索 +1 / 归还 +6"激励反向，导致线索池稀薄风险。新方案按"行为成本 + 网络贡献"双维度定价。

### 2. 24h 冷却期（详见 02）

双方确认归还后善行值不立即结算，留 24 小时反悔窗口。这是冒领的最后一道防线。

### 3. 防冒领的真实边界

私有特征只防陌生人冒领，防不住"代答 / 转述"。真实防御靠**线下交接 + 视频通话 + 疫苗本/旧照核验**。平台只能强引导，不能强保证。

### 4. 状态机三表（详见 02）

`lost_pets / found_pets / return_confirmations` 各有独立状态机，互相耦合的转换点必须显式写出来。

### 5. 一只宠物多人匹配

一只走失宠物可能被多人误认/捡到。`return_confirmations` 设计为 1:N，但只有一个能进 `confirmed`。

### 6. 数据来源合规

不爬取微博/小红书。冷启动数据来源：
- 开发者人工汇总本地寻宠群公开信息
- **必须**联系原发布者获得授权代发
- 标注"来源：XX 群 / 已获授权"
