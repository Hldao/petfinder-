# 05 · 合规风险与逐屏标注

> **范围**：本文基于微信小程序平台规则、《个人信息保护法》《数据安全法》《电信条例》等法律法规，针对 v5 高保真原型的 **16 屏** 进行逐屏合规标注。
>
> **目标**：让每一屏在开发前就明确"必须做什么、不能做什么、有哪些设计妥协"。

---

## 风险等级说明

| 等级 | 含义 | 处理 |
|------|------|------|
| 🔴 **P0** | 不做就上不了线 / 上线就被举报下架 | 开发期必须实现 |
| 🟠 **P1** | 影响产品形态的硬约束 | 设计阶段定方案 |
| 🟡 **P2** | 影响审核速度，但能解决 | 申请类目前准备 |
| 🟢 **P3** | 不踩雷，但要注意细节 | Code review 时检查 |

---

## 一、逐屏标注

### 1. `feed` — 信息流首页

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 位置精度泄露 | 🟠 P1 | 距离显示精度 ≥ 500m，**不要**展示精确经纬度 |
| 内容展示前已过审 | 🔴 P0 | 服务端确保推流前 `imgSecCheck` + `msgSecCheck` 已过 |
| 公益类目限制 | 🟡 P2 | 不能有广告位、推广位、商业入口 |

**设计建议**：卡片上"📍 500m" 显示的是**模糊距离**，不可点击跳转到精确地图。

---

### 2. `map` — 地图视图

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 用户位置敏感 | 🟠 P1 | 必须先 `requirePrivacyAuthorize` 再调用 `wx.getLocation` |
| 地图选型 | 🟡 P2 | 用**腾讯位置服务**（小程序集成最顺），不要用高德/百度 |
| 标点精度 | 🟠 P1 | 标点位置 = 实际位置 + 200-500m 随机偏移，**禁止** 1:1 还原 |
| 数据存储 | 🟡 P2 | 数据库存 geohash（精度 6-7 位）+ 加密，不存明文 lat/lng |

---

### 3. `detail-lost` — 寻宠详情

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 展示内容已过审 | 🔴 P0 | 服务端读取前再 check 一次审核状态（防止用户改了未重审） |
| 联系方式入口 | 🟠 P1 | **不可** 直接显示明文电话；点击"联系"跳 [[contact-guide]] 或走虚拟号 |
| 发布者信息 | 🟠 P1 | 只显示昵称 + 头像 + 善行值，**不显示**手机号/微信号 |
| 举报入口 | 🟡 P2 | 必须提供"举报"按钮（右上角"⋯"菜单内即可） |

---

### 4. `detail-found` — 招领详情

同 `detail-lost`。**额外注意**：

- 招领者上传的"宠物现状照片"如果含人脸（路人、收养人自己），P3 提示用户裁剪
- 不能展示"宠物当前位置"的精确点（保护招领人隐私，500m 模糊化）

---

### 5. `publish` — 发布表单 ⚠️ 核心 UGC 入口

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 文字内容审核 | 🔴 P0 | 提交时调 `security.msgSecCheck`，**未过不入库** |
| 图片内容审核 | 🔴 P0 | 上传后调 `security.imgSecCheck`，**未过删 COS 文件** |
| 联系方式收集 | 🟠 P1 | 手机号入库前加密（AES-256）；显示给他人时走虚拟号映射 |
| 位置收集 | 🟠 P1 | 收集前调 `requirePrivacyAuthorize('scope.userLocation')` |
| 表单字段 | 🟢 P3 | "悬赏金额"字段**禁止**，"奖励"字段须改为"心意感谢"（非货币） |
| 频率限制 | 🟡 P2 | 单用户每日发布 ≤ N 条（防恶意刷屏 / 引流广告） |

**设计建议**：审核失败时**只提示"内容包含不当信息"**，不回显具体哪个词命中（防绕过）。

---

### 6. `contact-guide` — 沟通建议

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 不能"引导平台外联系" | 🟠 P1 | 文案**禁止**出现"加微信""加 QQ""扫码加好友"等表述 |
| 联系方式实施 | 🟠 P1 | 推荐流程：弹"获取联系方式"→ 跳转**虚拟号拨号**或**站内 IM** |
| 隐私号成本 | 🟡 P2 | 腾讯隐私号约 0.x 元/分钟，按通话计费，需预算 |

**设计建议**：当前 v5 的"沟通建议"哲学好，但**必须配合虚拟号 / 站内 IM 才能真打通**——光建议没法用。

---

### 7. `return-checklist` — 归还自检

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 流程页无 UGC | 🟢 P3 | 无审核压力 |
| 法律免责声明 | 🟡 P2 | 文案中加"平台不参与核实，建议自行验证"等表述（已隐含） |

---

### 8. `return-success` — 归还成功

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 静态结果页 | 🟢 P3 | 无审核压力 |
| 善行值发放时机 | 🟠 P1 | 见 [[points-history]] 善行值规则；不能与现金挂钩 |

---

### 9. `thanks-edit` — 感谢信编辑

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 文字审核 | 🔴 P0 | `msgSecCheck` |
| 图片审核 | 🔴 P0 | `imgSecCheck`（合影常含人脸，敏感） |
| 合影隐私 | 🟢 P3 | 上传前提示"建议遮挡路人面部"，并要求"同意全平台展示"勾选 |
| 公开/私密选项 | 🟡 P2 | 必须提供"仅自己可见"选项 |

---

### 10. `messages` — 消息列表

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 站内 IM 入口 | 🔴 P0 | 列表项展示的"最后一条消息预览"也要走 [[chat]] 的审核结果 |
| 未读数 | 🟢 P3 | 无审核压力，常规设计 |

---

### 11. `chat` — 实时聊天 ⚠️ 高频 UGC

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 每条消息审核 | 🔴 P0 | 每条消息发送前 `msgSecCheck`，**未过不发送** |
| 图片消息 | 🔴 P0 | `imgSecCheck` + `mediaCheck`（异步），未过的图片用占位符 |
| 消息留存 | 🟡 P2 | 留存 ≥ 6 个月（电信条例 + 反诈要求） |
| "加微信"过滤 | 🟠 P1 | 服务端正则过滤"微信号 / wechat / wx / qq / 手机号"等，命中即拒 |
| 撤回机制 | 🟢 P3 | 可选，但需保留服务端原文用于审计 |

**设计建议**：消息体验上要做"温和拦截"，命中过滤时弹"建议在平台内沟通保护隐私"，而不是"违规"。

---

### 12. `profile` — 个人中心

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 隐私协议链接 | 🔴 P0 | 必须提供"用户协议"+"隐私政策"两个入口 |
| 账号注销入口 | 🔴 P0 | 必须提供"注销账号"功能，注销后**彻底删除**个人数据（30 天宽限期可保留） |
| 客服联系方式 | 🟡 P2 | 必须公开（电话或邮箱），位置在"设置"或"关于"里 |

---

### 13. `points-history` — 善行值流水

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 不能与现金挂钩 | 🔴 P0 | **禁止**充值、提现、积分兑商品、积分换券 |
| 不能拉新得分 | 🔴 P0 | **禁止**"邀请好友 +N"机制（被定性为分销） |
| 签到积分 | 🟠 P1 | 可以做，但频率 ≤ 每日 1 次，分数极低（防刷量嫌疑） |
| 等级 / 勋章 | 🟢 P3 | 鼓励做：等级、称号、徽章纯荣誉，最安全 |
| 兑公益捐赠 | 🟡 P2 | **可以做且加分**：积分换"为流浪猫捐 1 顿粮"，强化公益属性 |

**对照**：[[01-points-system]] 中的新规则（线索 +2+ 关键 +3 / 归还 +4）**合规**，无需调整。
~~原方案含「临时照料 +5」已随功能砍掉~~（详见 `docs/03-mvp-scope.md` §「关于临时照料的取舍」）。

---

### 14. `my-posts` — 我的发布

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 编辑后重审 | 🔴 P0 | 用户改了发布内容后，**必须重新过审**才能再展示 |
| 删除 / 下架 | 🟢 P3 | 用户可自行下架；删除后服务端保留 30 天审计期 |

---

### 15. `dispute` — 争议申诉

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 文字审核 | 🔴 P0 | `msgSecCheck` |
| 平台必须介入 | 🟡 P2 | 不能让用户"私聊解决"——必须有平台审核入口 |
| 证据留存 | 🟡 P2 | 申诉证据（截图、聊天记录）留存 ≥ 6 个月 |
| 处理时效公示 | 🟢 P3 | 文案明确告知"X 工作日内回复"，符合《消费者权益保护法》 |

---

### 16. `admin` — 后台审核 🚨 严重设计错误

**这一屏必须从小程序中完全剥离。**

| 风险点 | 等级 | 必做措施 |
|--------|------|----------|
| 小程序内禁止管理后台 | 🔴 P0 | 微信审核员一旦发现"管理员功能藏在用户端"立即驳回 |
| 隐藏入口也不行 | 🔴 P0 | 不能用"长按 logo 10 次进入"等彩蛋方式藏入口 |

**正确做法**：
- 单独做一个 **web 管理后台**（推荐：腾讯云开发的网页托管 + 同一个 CloudBase 数据库）
- v5 的 `admin` 屏可保留作**视觉/交互参考**，但开发时不能进小程序代码
- 移动端审核（如果需要）走"独立的管理员小程序"（独立 AppID，私密分享只给运营）

## 二、跨屏风险点视角

### A. UGC 审核接入点汇总

| API | 影响屏 |
|-----|--------|
| `security.msgSecCheck` | publish, chat, thanks-edit, dispute, my-posts (编辑) |
| `security.imgSecCheck` | publish, chat, thanks-edit |
| `security.mediaCheck` | chat (如支持视频), thanks-edit (如支持视频) |

**实施方案**：写一个 `contentSafety` 云函数封装三个 API，所有写入点统一调用。

### B. 联系方式合规链路

```
用户 A 发布 → 输入手机号 → 服务端 AES 加密入库
                              ↓
用户 B 点"联系" → 服务端创建虚拟号映射（A 真号 ↔ 虚拟号 X）
                              ↓
B 拨打 X → 隐私号平台转接到 A → 通话录音留存（可选）
```

涉及屏：publish (收集) → detail-lost / detail-found (展示) → contact-guide (引导) → chat (站内备选)

### C. 隐私授权时机

| 权限 | 申请时机 | 涉及屏 |
|------|----------|--------|
| `scope.userLocation` | 首次进 feed / map / publish | feed, map, publish |
| `scope.writePhotosAlbum` | 保存感谢信 / 分享图时 | thanks-edit, return-success |
| 相册（chooseImage）| 上传宠物照片 / 合影时 | publish, thanks-edit, chat |
| 手机号（getPhoneNumber）| 发布或登录时一次性获取 | publish 或登录页 |

启动时**必须**显示一次「隐私协议确认」（管理后台先配置好）。

### D. 数据留存合规

| 数据类型 | 最少留存 | 删除条件 |
|----------|----------|----------|
| 聊天记录 | 6 个月 | 用户注销后 30 天 |
| 发布内容（含已删除） | 30 天审计期 | 30 天后清理 |
| 申诉证据 | 6 个月 | 申诉关闭后 6 个月 |
| 用户基本信息 | 注销前持续 | 注销后立即匿名化 |

## 三、上线前 Checklist

### 开发阶段必做

- [ ] `contentSafety` 云函数封装三个 security API
- [ ] 服务端 AES 加密手机号 / 真实姓名等敏感字段
- [ ] 接入腾讯隐私号服务（或确认初版只用站内 IM）
- [ ] geohash + 位置偏移（200-500m）实现
- [ ] 隐私协议页 + 注销账号功能
- [ ] 编辑后重审逻辑
- [ ] 频率限制（单用户日发布上限）
- [ ] `admin` 从小程序代码库剥离，单独 web 后台

### 提审前必做

- [ ] 管理后台填写「用户隐私保护指引」
- [ ] 启动时展示隐私协议确认页
- [ ] 客服联系方式公开（profile / 设置内）
- [ ] 公益类目所需的合作证明（与大理本地救助站 / 动保协会）
- [ ] 备案：ICP 备案 + 公安备案（小程序也要）

### 运营准备

- [ ] 24 小时举报响应机制
- [ ] 内容审核员排班（人工复核机审误判）
- [ ] 用户协议 / 隐私政策法律审核（建议过一遍律师）

## 四、绝对不要碰的雷区

| 雷区 | 后果 |
|------|------|
| 悬赏金 / 现金打赏 / 积分兑现金 | 触发金融类目，立即下架 |
| 邀请好友得积分 / 分销机制 | 平台规则违规 + 反不正当竞争法 |
| 引导加微信 / QQ / 留电话 | 平台规则违规，封号高发 |
| 在小程序里塞管理员后台 | 审核驳回 |
| 用 Cloudbase 海外节点 | 数据出境违规 |
| 不接 UGC 审核 API | 一次举报即整改，反复触发即下架 |
