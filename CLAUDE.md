## 项目一句话定位

**寻宠 · 大理** — 本地化宠物走失/招领信息互助小程序。
**平台 = 撮合 + 关键节点 + 激励发放。不是留痕系统，不是流程管控者，更不是注意力收割机。**

主原型：`prototypes/app-prototype-v10.html` · 启动本地 server `python3 -m http.server 8765 --directory prototypes`

## 8 条核心哲学速查

只列标题，详情在 `docs/06-product-philosophy.md`。**每条都有真实决策案例支撑**。

1. **不挡用户的路** — 平台提供工具，不强制流程；用户在哪沟通不重要
2. **反冒领的真实边界** — 防御靠见面时的宠物反应 + 疫苗本 + 24h 冷却，**不靠身份信号**
3. **关键节点必须在平台内** — 善行值发放 / 归还确认 / 争议仲裁；非关键节点可在平台外
4. **WeChat 合规底线** — 不诱导加微信、不拉新返利、不撮合金钱交易
5. **大理冷启动的特殊性** — 城市尺度小 + 社群高度分层 + **真实优先于看起来繁荣**
6. **教育只说一次 · 不抢夺注意力** — 告知/介绍类元素显示一次后自动隐退
7. **不为焦虑的人增加心理负担** — 失主已焦虑 · UI 不叠加 · 想不到给希望的方案就不做
8. **真实 > 看起来繁荣** — 宁可 feed 空，也不要假数据/代发数据/刷的浏览量

## 评估新功能的 7 个判断问题

**任何"加功能"的建议在动手前必须问一遍**。任一项答案不理想 → 默认不做。

1. **工程**：长期维护成本？数据怎么更新 / 同步 / 清理？
2. **法律**：涉及个保法 / 微信合规 / 金融监管的哪些红线？最坏情况是什么？
3. **情感**：让失主**更焦虑还是更安心**？给希望还是制造压力？
4. **真实**：让数据更真实还是更"**看起来繁荣**"？
5. **运营**：上线后谁来持续维护？人力可持续吗？
6. **行为可执行性**（v10 r17 新增 · AI 协作高危陷阱）：用户在那个时刻**真的会做这个操作吗**？还是 AI 设计的"看起来合理"但用户在情绪驱动下不会执行的多步流程？详见 06 哲学 § 6.5「机制 vs 行为」
7. **视角对称性**（v10 r39 新增 · AI 协作高危陷阱）：产品里多个用户群体对这个功能/字段的**能力 / 信息差异**是什么？强信息群体（失主）vs 弱信息群体（拾主）的字段能力不同，强迫弱信息群体填会**产生假数据**（拾主瞎写"约 2 岁" → 失主跳过自家猫）。详见 06 哲学 § 6.6「多视角审视」

## Merge 安全 checklist（吸取真实事故教训）

> ⚠ 已发生过 3 次 merge 事故。**任何 merge 后必须做完这 3 步**：

### Merge 前
- `git fetch origin && git status -sb`
- `git log --oneline HEAD..origin/dadao` 看队友推了什么
- `git diff --stat HEAD..origin/dadao` 看影响规模

### Merge 后必做
- [ ] **函数完整性 grep**：刚删的函数名不能被新代码引用 · 刚加的函数不能没被引用
- [ ] **浏览器跑核心流程** + DevTools Console **不能有 ReferenceError / TypeError**
- [ ] **commit message** 明确写"合并自 XX commit，处理了 YY 冲突"

### 冲突解决三问
遇到 `<<<<<<<` 时对每个冲突块问：
1. 「他们的代码引用了我们删掉的东西」？→ 要么恢复，要么改调用，**绝不能"双方都接受"**
2. 「他们删了我们的函数 + 没引用」？→ 安全接受删除
3. 「双方都改了同一段」？→ 必须人工理解双方意图后合并

## AI 协作者特别注意

如果你是 AI（Claude / 其他）：

1. **不要根据"行业默认"做产品建议** — 这个产品反主流（不抢注意力 / 不制造焦虑 / 不假装繁荣）
2. **加功能前必须查奥卡姆反例库** — 16+ 条里大概率已经覆盖你想加的东西
3. **跑过 7 个判断问题再动手** — 任一项不理想就停
4. **改 prototype 用 Edit 工具，禁用 sed**
5. **每个独立改动立即 commit + push** — 不要攒着
6. **遇到 merge 冲突时**：函数定义和调用必须配对，**不能"我删它调用"或"它删我函数"**
7. ⚠ **不要为了"让用户觉得有用"而捏造无法追踪的数据**
   AI 协作时容易被"温暖体验"语言带跑，把好的形式套到伪数据上。
   **判断标准**：每个数字 / 每个具体描述都要能回答「这数据真实从哪来？」
   - ✅ 浏览量 = 小程序埋点（可追踪）
   - ✅ 分享次数 = `wx.shareAppMessage` 回调（可追踪）
   - ❌ "古城北门的咖啡馆转给了 3 位常客"（**全部捏造**：身份/名字/传播链都不可追踪）
   - ❌ "喜洲的客栈群有人帮你转了一次"（微信明确禁止追踪转发目的地）

   答不出来 = 别写。**真实 > 看起来繁荣**（第 8 条）。
8. **「解释 = 唤起」原则**（第 6 条延伸）：用户没意识到的问题，不要主动"解释清楚"——你的解释本身就是提醒
   - ❌ "你的手机号永远不会公开"（本来用户没在想手机号）
   - ✅ 默认就不收集 → 文案完全不提
